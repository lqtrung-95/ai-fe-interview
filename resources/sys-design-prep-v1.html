⚡ System Design Principles

Buổi 1 — Nền tảng lý thuyết. Đây là câu hỏi đầu tiên interviewer sẽ hỏi.

CAP Theorem

Must Know

Một hệ thống phân tán không thể đồng thời đảm bảo cả 3: Consistency, Availability, Partition Tolerance. Chỉ được chọn 2 trong 3. Trong thực tế, Partition Tolerance là bắt buộc → luôn chọn giữa CP hoặc AP.

CP Systems

Khi có network partition → từ chối phục vụ để giữ consistency. Banking, ZooKeeper, etcd, HBase.

AP Systems

Khi có network partition → vẫn phục vụ nhưng có thể return stale data. Cassandra, DynamoDB, CouchDB.

📌 Interview tip: Đừng chỉ nói "strong" hay "eventual". Hỏi ngược: "Use case này cần read-your-writes không?" — điều này cho thấy hiểu sâu.

ACID vs BASE

Database

ACID — SQL Databases

  • Atomicity: All-or-nothing transaction
  • Consistency: DB luôn ở trạng thái valid
  • Isolation: Concurrent transactions không ảnh hưởng nhau
  • Durability: Committed data không mất kể cả khi crash
MySQLPostgreSQLOracle

BASE — NoSQL Databases

  • Basically Available: Luôn phục vụ dù có thể lỗi nhỏ
  • Soft state: State có thể thay đổi theo thời gian
  • Eventually consistent: Cuối cùng tất cả replicas sẽ đồng bộ
CassandraDynamoDBCouchDB

Database Sharding

Scaling

Sharding chia data ra nhiều database servers để scale horizontally. Mỗi shard chứa một phần dataset.

StrategyCách hoạt độngPros / Cons
Hash-basedshard = hash(key) % N✅ Phân phối đều ❌ Khó thêm shard
Range-basedA-M → shard 1, N-Z → shard 2✅ Dễ range query ❌ Hotspot risk
Consistent Hashing ★Nodes trên ring, key → nearest node theo clockwise✅ Thêm/bớt shard chỉ move K/N keys
⚠️ Đừng shard sớm! Thứ tự scaling: Index → Cache → Read Replicas → Vertical → Sharding. Hầu hết apps không cần sharding đến 50M+ users.

Master-Slave Replication

HA
  • Tất cả writes đi qua Master. Master ghi binlog → slaves replicate async.
  • Tất cả reads phân phối cho Slaves → giảm tải Master (read:write thường 10:1)
  • Master fail → promote Slave lên Master mới (auto với MHA/Orchestrator)
  • Replication lag: Slave có thể trả stale data trong vài ms-giây. Cần read-your-writes → đọc từ Master.

Database Indexing

Performance
-- Không có index: full scan O(n)
SELECT * FROM users WHERE email = 'alice@example.com';

-- Tạo index: O(log n)
CREATE INDEX idx_users_email ON users(email);

-- Composite index — leftmost prefix rule!
CREATE INDEX idx_posts ON posts(user_id, created_at DESC);
-- OK: WHERE user_id=? ORDER BY created_at DESC
-- NOT OK: WHERE created_at=?  (skips leftmost column)

-- ❌ Function trên indexed column → không dùng được index
SELECT * FROM orders WHERE YEAR(created_at) = 2024;
-- ✅ Dùng range query thay thế
SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31';
⚠️ Index có chi phí: write chậm hơn, tốn storage. Chỉ index columns trong WHERE, JOIN ON, ORDER BY thường xuyên.

ConsistentWrite latency tăng

Write-Back (Write-Behind)

Write cache → ACK ngay. Async write to DB. Nhanh nhất nhưng risk mất data nếu cache crash.

Fast writesData loss risk

Cache Problems — Pitfalls cần tránh

Common Issues

🔥 Cache Stampede

Key expire → nhiều requests cùng miss → hammer DB.
Fix: Mutex lock hoặc jitter TTL (TTL = base + random(0,300)).

🕳️ Cache Penetration

Request key KHÔNG TỒN TẠI trong DB → bypass cache liên tục.
Fix: Cache null values hoặc Bloom Filter.

💥 Cache Avalanche

Nhiều keys expire CÙNG LÚC → DB spike.
Fix: Jitter: TTL = base + random(0, 300).

🌊 Stream Processing

Buổi 3 — Kafka, message queues, xử lý event realtime

Message Queue vs Event Streaming

Concepts

Message Queue (RabbitMQ, SQS)

  • Messages được consume một lần rồi xóa
  • Point-to-point hoặc pub/sub
  • Tốt cho: task queues, email, job scheduling
  • Không replay được

Event Streaming (Kafka)

  • Messages được lưu lâu dài (7+ ngày)
  • Nhiều consumer groups đọc độc lập
  • Tốt cho: event sourcing, audit log, analytics
  • Có thể replay từ đầu

Kafka Core Concepts

Architecture
ConceptGiải thích
TopicKênh nhận/gửi messages. Ví dụ: "user-clicks", "orders", "notifications"
PartitionChia topic thành segments. Messages trong cùng partition có ordering đảm bảo. Nhiều partitions = parallelism cao hơn.
OffsetID tuần tự của message trong partition. Consumer track offset đã đọc đến đâu.
Consumer GroupNhiều consumers cùng group chia nhau đọc partitions. Mỗi partition chỉ đọc bởi 1 consumer trong group.
Replication FactorSố bản copy. RF=3 → 1 leader + 2 followers. Leader fail → follower promote.
📌 Ordering: Chỉ đảm bảo trong cùng partition. Muốn related events theo thứ tự → dùng cùng partition key (vd: user_id làm key → tất cả events của user → 1 partition).

API Gateway

Architecture

API Gateway là entry point duy nhất cho tất cả client requests, thực hiện cross-cutting concerns tập trung:

  • Authentication/Authorization: Verify JWT/API Key trước khi forward
  • Rate Limiting: Giới hạn requests per user/IP/API key
  • Request routing: /api/users → User Service, /api/orders → Order Service
  • SSL Termination: Decrypt HTTPS một lần
  • Load balancing: Distribute across service instances
  • Caching: Cache responses cho repeated identical requests
  • Circuit breaking: Fail fast khi downstream quá tải
  • Observability: Centralized logging, metrics, tracing

🗂️ Interview Framework

Buổi 6 — 4 bước tiếp cận. Đây là điều quyết định pass/fail.

The 4-Step Framework

Must Follow
1
Clarify Requirements — 5 phút

Functional (What to build): Core features, User types, Data patterns (read/write heavy?)

Non-Functional (How well): Scale (users/QPS), Latency (p99 target), Availability (99.9% vs 99.99%?), Consistency (strong vs eventual?)

2
Capacity Estimation — 5 phút
Write QPS = Total requests per day / 86,400
Storage = # items × avg item size × retention period
Bandwidth = QPS × avg response size
3
High-Level Design — 15 phút
  • Identify components: Client, API Gateway, Services, DB, Cache, CDN, Queue
  • Design APIs: endpoint, method, request/response schema
  • Data model: tables/collections chính, relationships
  • Describe data flow cho core use cases
4
Deep Dive — 20 phút
  • Bottlenecks: reads nhiều → cache; writes nhiều → queue; hot data → sharding
  • Scale từng component
  • Edge cases: server failure, network partition, thundering herd
  • Trade-offs: Consistency vs Availability, Latency vs Throughput

Estimation Cheat Sheet

Math

Time conversions

1 day  = 86,400 sec ≈ 10⁵
1 month ≈ 2.5M sec
1 year  ≈ 30M sec

Typical object sizes

Tweet/short text  ≈ 300 bytes
User profile row  ≈ 1 KB
Thumbnail image   ≈ 200 KB
HD photo          ≈ 3 MB
1 min video       ≈ 10 MB (compressed)

Twitter example (100M tweet/day)

Write QPS: 100M / 86,400 ≈ 1,160/s
Read QPS (10:1): ≈ 11,600/s
Peak (3×): ≈ 35,000/s

Storage/day: 100M × 300B = 30 GB
Storage/year: 30GB × 365 ≈ 11 TB

Media (10% tweets): 10M × 200KB = 2TB/day

Top 10 Lỗi Thường Gặp

Avoid!
  • Bắt đầu code ngay mà chưa clarify requirements
  • Over-engineer từ đầu — start simple, scale when needed
  • Bỏ qua estimation — không biết scale → design sai
  • Không nói trade-offs — interviewer muốn nghe đánh đổi
  • Quên SPOF — LB, DB, cache đều có thể là SPOF
  • Không hỏi consistency requirements — ảnh hưởng lớn đến design
  • Thiếu data model — HLD không có schema là chưa hoàn chỉnh
  • Bỏ qua security — auth, rate limiting, encryption
  • Không có monitoring — metrics, logs, alerts là mandatory
  • Nói nhiều, vẽ ít — diagram > 1000 words

Buổi 7 — Design a URL shortening service (bit.ly / tinyurl.com)

Requirements & Estimation

Step 1-2

  • Redirect short URL → original URL
  • Custom alias, URL expiration
  • Analytics: click count, geo, device

Key Design Decisions

Deep Dive

// ID → base62 encode
const CHARS = '0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ';
function encode(id) {
  let result = '';
  while (id > 0) { result = CHARS[id % 62] + result; id = Math.floor(id / 62); }
  return result.padStart(7, '0'); // "1LY7VK0"
}

301 Moved Permanently

Browser cache redirect → click sau không qua server. Tiết kiệm bandwidth nhưng không track được analytics.

302 Found (Temporary) ★

Browser không cache → mỗi click đều qua server. Track analytics được. Dùng cho bit.ly-like services.

💡 Analytics: Thay vì UPDATE view_count += 1 (write contention), dùng: click event → Kafka → Flink aggregate → write summary mỗi phút.

Buổi 9 — Facebook/Instagram/Twitter — News feed, fanout

Fanout Problem — News Feed

Hard Problem

Fanout on Write (Push)

Khi post → write vào ALL followers' feed.

  • Read cực nhanh (pre-built)
  • Celebrity 10M followers = 10M writes/post!
  • Waste storage

Fanout on Read (Pull)

Khi load feed → fetch từ all followees.

  • Celebrity ok (ít writes)
  • Read chậm, complex merging
  • 1000 followees = 1000 queries

🔀 Hybrid (Twitter/Facebook) ★

  • Regular users (<1M followers): Fanout on Write. Pre-built feed cached Redis.
  • Celebrities (>1M followers): Fanout on Read.
  • Khi load feed: đọc pre-built + fetch celebrity posts + merge by timestamp.

Snowflake ID — Message Ordering

ID Generation

64-bit integer: 41 bits timestamp + 10 bits machine_id + 12 bits sequence

  • Time-sortable: sort by ID = sort by time → không cần created_at để sort
  • Globally unique: không cần central coordinator
  • High throughput: 4096 IDs/ms per machine
  • 64-bit int: nhỏ và fast hơn UUID (128-bit)
// Group chat schema (Cassandra)
CREATE TABLE group_messages (
  group_id    UUID,
  message_id  BIGINT,     -- Snowflake ID (sortable!)
  sender_id   UUID,
  content     TEXT,
  created_at  TIMESTAMP,
  PRIMARY KEY ((group_id), message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);
