🌳 React Reconciliation Decision Tree

How React decides what to update, mount, or unmount.

New Virtual DOM Tree
Compare to previous tree at same position
↓ for each element
Element type changed?
e.g., <div> → <span>, or <Foo> → <Bar>
YES → Unmount + Remount
State LOST · child tree destroyed · effects cleaned up
NO → Same type
Continue checking props...
↓ if same type
Props changed?
NO → Skip update
Reuse existing instance, no re-render needed
YES → Update props
Same instance, state preserved, effects may re-run
⚠ Killer trap: Defining a child component INSIDE a parent creates new function ref every render → React sees different "type" → unmount+remount each time → state resets! 
function Parent() {
  // ❌ BAD: Child gets fresh identity each render
  const Child = () => <div>...</div>;
  return <Child />;
}
💡 The key prop: Helps React track elements in lists. Stable IDs (not array index!) avoid unnecessary remounts.

🤔 useState vs useReducer Decision Flow

Need state management
↓
Q1: Multiple interdependent fields? e.g., form with amount + currency + recipient
↓ YES
Q2: Next state depends on previous via complex logic? e.g., toggle with side effects, undo/redo
↓ YES
Q3: Many distinct actions update state? CRUD: SELECT, ADD, REMOVE, RESET, SUBMIT
↓ ANY YES
✅ useReducer
Pure reducer, testable separately. Dispatch flows down tree cleanly.
All NO? → useState
Simpler, less ceremony.
💡 Bonus pattern: Pair useReducer + Context for global-ish state without Redux. Lightweight Redux-like architecture.

🔄 useEffect Lifecycle & Cleanup

When does cleanup actually run? Critical for avoiding memory leaks.

Component mounts
Effect runs first time · returns cleanup fn
↓
Deps change
1. Run cleanup fn (from previous effect)
2. Run new effect
↓
Deps change again
Cleanup → new effect (repeat)
↓
Component unmounts
Final cleanup runs, no new effect
⚠ Forget cleanup = memory leak, stale state, "setState on unmounted component" warning

When You MUST Cleanup

📡 Event subscriptions
DOM, WebSocket, EventEmitter · use removeEventListener
⏱ Timers
setInterval, setTimeout · use clearInterval/clearTimeout
🌐 Async fetches
Use AbortController · prevent setState after unmount
🎬 Observers
IntersectionObserver, MutationObserver · call .disconnect()

🧮 useMemo vs useCallback

useMemo
Caches:Computed VALUE
Returns:Result of the function
Use for:Heavy compute, derived state
const filtered = useMemo(
  () => items.filter(i => i.active),
  [items]
);
useCallback
Caches:Function REFERENCE
Returns:The function itself
Use for:Stable refs for memo'd children
const handleClick = useCallback(
  () => doSomething(id),
  [id]
);
⚠ Don't over-use: Memoization has cost (memory + comparison). Default React re-renders are usually fast enough.
💡 Rule: Profile first. Add memoization only when there's a measurable problem (slow re-render, child wrapped in React.memo that keeps re-rendering due to new prop ref).

⏱ Promise Combinators Comparison

Promise.all
✅ All resolve → array of values
❌ ANY reject → reject immediately
Use: Parallel fetches that all must succeed.
Promise.allSettled
Always resolves → array of {status, value/reason}
Use: Independent fetches, you want all results regardless.
Promise.race
First to SETTLE (resolve OR reject) wins
Use: Timeout pattern · which CDN responds first.
Promise.any
First to RESOLVE wins
All reject → AggregateError
Use: Fallback pattern · any RPC works.

📈 Exponential Backoff Retry Pattern

Critical for payment SDK reliability against transient network failures.

1 Try → FAIL → wait 1s + jitter → 2 Try → FAIL → wait 2s + jitter → 3 Try → FAIL → wait 4s + jitter → 4 Try → SUCCESS ✓
💡 Formula: delay = baseDelay * 2^attempt + random(0, 100ms)
Jitter is essential - prevents thundering herd when many clients retry simultaneously after a server recovery.
⚠ Don't retry these: 4xx errors (user bug), idempotency violations, "tx already submitted" (could double-pay).

🎯 Event Delegation Pattern

❌ Per-element listeners
items.forEach(item => {
  item.addEventListener(
    'click', handler
  );
});
❌ O(N) listeners
❌ Memory leak on item removal
❌ Dynamic items need rebinding
✅ Delegation (1 listener)
container.addEventListener(
  'click', (e) => {
    const item = e.target
      .closest('[data-id]');
    if (item) handle(item);
  }
);
✅ O(1) listener
✅ Auto-handles dynamic items
✅ Single source of truth
💡 Fun fact: React does this automatically. onClick is NOT a native DOM event - it's React's synthetic event delegated at the root.

👛 EOA vs Smart Contract Wallet

Core distinction in crypto wallets. Fun.xyz builds smart wallets.

🔑 EOA (Externally Owned Account)
Control:Private key
Examples:MetaMask, hardware wallets
Gas:Must pay in ETH
Logic:Fixed (sign + send only)
Recovery:Seed phrase only
Batch tx:❌ Not possible
📜 Smart Contract Wallet
Control:Contract code logic
Examples:Safe (Gnosis), Argent, Fun.xyz
Gas:Paymaster can sponsor
Logic:Custom: multisig, sessions, limits
Recovery:Social recovery, guardians
Batch tx:✅ Approve+swap in one tx
💡 Fun.xyz angle: Smart wallets enable gas abstraction (user pays USDC, not ETH), batch transactions, and social recovery - exactly the UX wins they're pitching.

⛽ Gas Fee Breakdown (EIP-1559)

Why a failed tx still costs gas. Critical for SDK error handling.

Gas Limit
Max gas units the user authorizes (safety ceiling)
set by user
Max Fee Per Gas
Upper bound user is willing to pay per gas unit
upper bound
Base Fee
Set by network demand · BURNED (removed from supply)
network
Priority Fee (Tip)
Paid to validator · incentive to include your tx
validator
Gas Used
Actual computation cost (≤ Gas Limit)
actual
⚠ Why failed tx still costs gas: EVM consumed computational work before hitting the revert. Validator must be paid. State rolled back, gas not refunded.
💡 SDK implication: Always estimateGas + eth_call simulate BEFORE asking user to sign. Catches reverts without burning user money.

🚨 Error Handling Architecture for Payment SDK

6 error categories, each with specific UX response.

🟡 User Error (4xx)
Invalid input, wrong chain, insufficient balance.
UX: Inline form error + actionable fix.
📡 Network Error
Connection lost, RPC down.
UX: Auto-retry with backoff, fallback RPC.
👛 Wallet Error
User rejects sign, locked wallet.
UX: Toast with guidance.
⛓ Chain Error
Tx revert, out of gas, slippage.
UX: Simulate BEFORE sign, show clear reason.
🐛 SDK Bug
Misconfig, unexpected state.
UX: Generic message + Sentry log with context.
🔥 Provider Error
Fun.xyz backend down.
UX: Status page link, retry.

How Proxy Sandbox Works

// Simplified Qiankun internal
const proxyWindow = new Proxy(window, {
  get(target, key) {
    return sandboxGlobals[key] || target[key];
  },
  set(target, key, value) {
    sandboxGlobals[key] = value;  // goes to sandbox, not real window
    return true;
  }
});
💡 Detection flag: window.__POWERED_BY_QIANKUN__ === true tells micro-app "you're in a sandbox" - useful for conditional behavior.
