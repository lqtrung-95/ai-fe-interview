## Key Insights

- Supabase Auth UUIDs must be bridged to Prisma `User.id` on first login — pick a strategy now (webhook vs. Server Action) and stay with it.
- **Content already exists in `resources/`** — four HTML files written by user, EN-translated and extracted into seed bank instead of hand-curating. `fe-prep.html` is literally a structured JS Q-array (`DATA_META`, `DATA`); the rest are deep guides yielding `expectedPoints` rubric content.
- Extraction is a one-off script run during seeding, output committed to `prisma/seed/questions/*.json`. No runtime extractor.
- Vietnamese source → English via batch LLM translation in the extraction script. No `language` field on `SeedQuestion`; product is EN-only.
- fun-xyz-prep.html topics fold into existing React/JavaScript/Performance buckets; do NOT ship as a company-named pack at MVP.
- shadcn/ui uses Tailwind v4 with `@theme` directive — initialize the design tokens early (calm/focused palette per PRD §12.1).
- Use `pnpm` for workspace consistency and faster installs.

**Non-functional**

## Architecture

- Next.js App Router with `(marketing)` and `(app)` route groups.
- Supabase clients split server vs browser per `@supabase/ssr` docs.
- Prisma uses `DATABASE_URL` pointing to Supabase pooler; `DIRECT_URL` for migrations.
- Seed runs via `prisma db seed` reading from `prisma/seed/questions/*.json`.

## Implementation Steps

1. `pnpm create next-app@latest . --typescript --tailwind --app --src-dir --no-eslint` then add ESLint config manually.
2. Install: `prisma @prisma/client @supabase/supabase-js @supabase/ssr zod`.
3. Set up Supabase project, copy URL + anon + service-role keys to `.env.local`.
4. Initialize Prisma: `pnpm prisma init`. Paste schema from [system-architecture.md §6](../../docs/system-architecture.md#6-data-model-prisma).
5. Run `pnpm prisma migrate dev --name init`.
6. Implement Supabase server/browser clients per `@supabase/ssr` patterns.
7. Implement OAuth callback route + middleware for session refresh.
8. Implement webhook that creates `User` row on Supabase `auth.users` INSERT.
9. Add shadcn: `pnpm dlx shadcn@latest init`. Pick Slate base. Add components: `button card input label select tabs toast`.
10. Build landing page per PRD §7.1 — hero, benefits, supported categories, CTA.
11. Build onboarding form (PRD §7.2) — 5 questions, saves to `User` table.
12. Build topic selection (PRD §7.3) — multi-select chips by topic with difficulty filter.
13. **Extraction script** — `scripts/extract-seed-from-html.ts`:
    a. Parse `resources/fe-prep.html` → grab `DATA_META` + `DATA` JS arrays via regex/JSDOM → map `{cat, level, q, a?}` to `SeedQuestion` shape (topic from `cat`, difficulty from `level`).
    b. Parse `resources/fe-prep-2.html` + `resources/sys-design-prep-v1.html` → walk `<h2>`/`<h3>` sections → use LLM to generate 2-3 questions per section with `expectedPoints[]` derived from the section's prose.
    c. Parse `resources/fun-xyz-prep.html` → extract React/JS/Web3 conceptual questions, label as React or JavaScript topic (strip company branding).
    d. Batch-translate all VN content to EN via OpenAI; cache translations on disk (`scripts/.translation-cache.json`) so re-runs are cheap.
    e. Deduplicate semantically similar questions (LLM judge or simple normalize-and-compare).
    f. Emit per-topic JSON files: `prisma/seed/questions/{javascript,react,system-design,performance,browser-apis,testing,behavioral}.json`.
14. Manual QA pass — eyeball ~10% of extracted questions for translation quality and `expectedPoints` sanity.
15. Implement `prisma/seed.ts` — reads all JSONs, upserts into `SeedQuestion` (idempotent by `id`).
16. Add `pnpm seed` + `pnpm extract-seed` scripts in `package.json`.
17. Run extraction → run seed → verify rows in Supabase Studio (target 60-100 questions).
18. Commit `prisma/seed/questions/*.json` to repo (translation cache too, so future re-runs don't re-spend tokens).
19. Run `pnpm tsc --noEmit` and fix all errors.

## How to finish Phase 01

1. Wait for Groq daily TPD reset OR set `LLM_SMART_PROVIDER=openai` in `.env.local`.
2. `pnpm extract-seed` — disk cache preserves the 25 already-extracted Qs; rest will fill in.
3. `pnpm seed` — upserts into Supabase `SeedQuestion`.
4. Eyeball `prisma/seed/questions/*.json` for translation quality on top ~10%.

## Risk Assessment

| Risk | Mitigation |
|---|---|
| LLM translations lose technical nuance (e.g. "stale closure") | Use glossary prompt with locked terms (English-preserved); manual QA on top 10% |
| Extraction script picks up irrelevant content from prose files | Constrain to `<h2>`/`<h3>` sections only; LLM prompt explicitly rejects non-question content |
| Translation token cost | Cache by source-text hash; one-time spend (~$2-5 estimated for all 4 files) |
| Duplicate questions across files | Dedupe pass — normalize, then LLM judge for semantic similarity |
| fun-xyz Web3 content too niche for general FE audience | Tag Web3 questions; surface only when user selects "Crypto/Web3" company type in onboarding |
| Supabase webhook latency causes race on first login | Fallback: also create `User` row in Server Action on first authenticated request if missing |
| Tailwind v4 + shadcn version mismatch | Use shadcn `canary` if needed; pin Tailwind version |

## Security Considerations

- Service role key never exposed to client.
- Webhook validates Supabase signature header.
- Middleware refreshes session cookies on every Server Component request.
